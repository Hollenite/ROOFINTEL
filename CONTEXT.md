## 4. Five-Stage Pipeline

### Overview

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  1. DATA     │──▶│  2. MODEL    │──▶│  3. VECTOR   │──▶│  4. SOLAR    │──▶│  5. UI +     │
│  LOAD        │   │  INFERENCE   │   │  -IZE        │   │  ESTIMATE    │   │  ENRICH      │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
GeoTIFF / synth    U-Net → binary     mask → Shapely      kW / kWh per        Streamlit +
+ CRS/transform    mask (optional)    polygons + area      roof + aggregate    policy/market/
                                      in m²                                    confidence/
                                                                               subsidy
```

### Stage 1 — Data Load (`src/data.py`)
- `load_image(path)` reads any GeoTIFF via rasterio. If the file has no CRS, a projected CRS (EPSG:32643 — UTM 43N, Pune region) is synthesized with a 0.3 m/pixel GSD so downstream area calculations work.
- `generate_synthetic_tile()` creates a fake 256×256 RGB image + binary mask + real UTM transform + EPSG:32643. No external data needed.
- Both return `(image, transform, crs)` — the same tuple shape.

### Stage 2 — Model Inference (`src/infer.py`) — optional
- Only runs in "Demo Mode B" or when a checkpoint exists. In the current Streamlit app, inference is not wired to the Run Pipeline button — the app uses pre-existing masks or ground-truth labels.
- `predict_mask(model, image)` → resizes to 256×256, runs U-Net forward pass, thresholds at 0.5, upscales back to original resolution using nearest-neighbour.

### Stage 3 — Vectorization (`src/vectorize.py`)
- `mask_to_polygons(mask, transform, crs, ...)` → `List[Dict]`
- Each dict: `{"geometry": Shapely Polygon, "area_value": float, "area_unit": str, "pixel_area": float}`
- Uses `rasterio.features.shapes()` to extract polygons from binary mask.
- Area is in m² when CRS is projected, otherwise pixel² with warning.
- Results sorted largest-first. Small polygons filtered by `min_area`.

### Stage 4 — Solar Estimation (`src/estimate.py`)
- `estimate_all_roofs(polygons, config)` → `(per_roof: List[Dict], aggregate: Dict)`
- See §8 for exact math.

### Stage 5 — UI + Enrichment (`app.py`)
The app adds a v2 enrichment pass after vectorization:
```python
polygons = enrich_polygons_with_feasibility(polygons, image, mask)   # src/feasibility.py
per_roof, aggregate = estimate_all_roofs(polygons, config)            # src/estimate.py
tile_conf = compute_tile_confidence(...)                               # src/confidence.py
# policy_records and market_records loaded from JSON at startup       # src/policy.py / market.py
per_building_savings = [_compute_solar_savings(...) for r in per_roof]  # app.py inline
```
All results stored in `st.session_state["results"]` so Streamlit download buttons don't trigger recomputation.

## 7. Solar Estimation Math

### Formula

```
usable_area   = roof_area_m2 × roof_usability_factor          (default: 0.65)
system_kw     = usable_area  × panel_power_density_kw_per_m2  (default: 0.18 kW/m²)
monthly_kwh   = system_kw    × monthly_generation_kwh_per_kw  (default: 110 kWh/kW)
annual_kwh    = monthly_kwh  × 12
             OR system_kw    × annual_generation_kwh_per_kw    (if override set)
```

### The performance_ratio Field — Authoritative Clarification

`SolarConfig.performance_ratio = 0.78` exists in the dataclass but is **NOT multiplied** in `estimate_single_roof`. This is **intentional and correct**.

`monthly_generation_kwh_per_kw = 110` is a **delivered** value that already bakes in:
- Solar irradiance (peak sun hours ~4.5 h/day for India average)
- System performance ratio (0.78)
- Inverter efficiency
- Wiring and soiling losses
- Temperature derating

The field is retained in SolarConfig for transparency (displayed in the text report) and to allow future refactoring if the formula is changed to use raw irradiance × PR explicitly.

The `AUDIT_REPORT.txt` from Feb 2026 incorrectly flagged this as a bug. It has since been documented in README.md, PROJECT_ANALYSIS.txt, and SolarConfig docstrings.

### PM Surya Ghar Subsidy (app.py `_compute_solar_savings`)

| Capacity | Central Govt | State Govt (Maharashtra) | Total |
|----------|-------------|--------------------------|-------|
| 1 kW | ₹30,000 | ₹15,000 | ₹45,000 |
| 2 kW | ₹60,000 | ₹30,000 | ₹90,000 |
| 3–5 kW | ₹78,000 (cap) | ₹30,000 | ₹1,08,000 |

Installation cost assumed: ₹67,000/kW. Payback = net_cost / annual_saving. Lifetime = (annual_saving × 25) − net_cost.

## 8. ML Model Details

| Parameter | Value |
|-----------|-------|
| Architecture | U-Net |
| Encoder | ResNet34 (ImageNet pretrained) |
| Library | segmentation-models-pytorch (SMP) |
| Input channels | 3 |
| Input size | 256 × 256 (resized from original) |
| Output | 1-channel sigmoid logit → binary mask |
| Parameters | ~24M |
| Loss | DiceBCE: `0.5 × BCE + 0.5 × (1 − Dice)` |
| Optimizer | Adam, lr=1e-3 |
| Batch size | 4 |
| Epochs | 10 |
| Dataset | WHU Satellite Dataset I — 204 tiles, 512×512, global cities |
| Split | 80/20 train/val |
| Best checkpoint | Val Dice = 0.7166 (epoch 7) → `checkpoints/best.pth` |
| Training device | CPU (no GPU available during development) |
| Training time | ~50 min on CPU for 10 epochs |

### Normalization (consistent train ↔ infer)
Both `SpaceNetDataset.__getitem__` (train.py) and `predict_mask` (infer.py) normalize via:
```python
if img.max() > 1.0:
    img = img / 255.0
```
No ImageNet mean/std normalization is applied. Both paths are consistent.

### Inference flow in predict_mask
1. Ensure 3-band input
2. PIL resize to 256×256 (BILINEAR)
3. Divide by 255 → [0,1] float
4. Run forward pass → sigmoid → threshold at 0.5
5. PIL resize back to original resolution (NEAREST — preserves edges)

## 12. Code Quality Audit

> Updated May 2026. Supersedes AUDIT_REPORT.txt (Feb 2026).

### Confirmed Issues in Current Code

#### HIGH severity

**1. Zero automated tests**
- No test files exist anywhere in the repository
- Functions like `compute_area_m2`, `estimate_single_roof`, `mask_to_polygons` are untested
- Any refactor risks silent breakage
- Fix: `pytest` + unit tests for the 5–6 core math functions; 2–4 hours effort

**2. No error handling in app.py pipeline** (`app.py:679–891`)
- The entire pipeline (vectorize → feasibility → estimate → confidence → render) runs inside `if run:` with no try/except
- Any exception produces a raw Python traceback in the Streamlit UI
- Fix: wrap pipeline in `try: ... except Exception as e: st.error(f"Pipeline failed: {e}")` blocks

**3. No ML model evaluation script**
- No script computes IoU, Dice, precision/recall on held-out tiles
- Val Dice 0.7166 was reported during training but cannot be independently reproduced
- Fix: `scripts/evaluate.py` running `validate()` from train.py on a test split

#### MEDIUM severity

**4. `discover_pairs` pairs by count, not filename** (`train.py:191–195`)
```python
image_paths = image_paths[:min_n]
mask_paths  = mask_paths[:min_n]
```
If `images/` and `masks/` have different sort orders (e.g. OS-level glob differences), wrong image-mask pairs are silently used. Fix: pair by stem name match.

**5. `_draw_polygon` ignores interior rings** (`viz.py:134–138`)
```python
coords = np.array(geom.exterior.coords)   # holes ignored
```
Building polygons with courtyards or holes will render incorrectly — the hole fills in solid. Fix: loop over `geom.interiors` and draw each ring with background fill.

**6. No data augmentation in training** (`train.py`)
- `SpaceNetDataset` applies no random flips, rotations, or color jitter
- With only 204 training tiles, the model will overfit
- Fix: add `torchvision.transforms` or `albumentations` augmentation pipeline

**7. Hardcoded paths throughout**
- `app.py:615`: `data_dir = Path("data/raw")`
- `train.py:208`: `--data_dir` default `"data/raw"`
- `train.py:213`: `--checkpoint_dir` default `"checkpoints"`
- Fix: environment variables (`DATA_DIR`, `CHECKPOINT_DIR`) with Path fallbacks

**8. No logging framework**
- All modules mix `print()` and `warnings.warn()` with no log levels
- In production, this produces stdout noise with no filtering ability
- Fix: `logging.getLogger(__name__)` in each module; replace print/warn calls

#### LOW severity

**9. Location key lookup fragility** (`app.py:556–568`)
```python
display_names = [rec.location_name ... ]        # e.g. "Los Angeles"
selected_location_key = selected_location_display.lower()  # "los angeles"
```
Baselines are keyed by `location_name.lower()`. If a display name is edited in the JSON without updating the lookup, `policy_records.get(selected_location_key)` returns None silently. No error surface.

**10. Massive inline CSS** (`app.py:76–394`, 318 lines)
- CSS occupying 26% of app.py makes the file hard to read and maintain
- Fix: extract to `static/app.css` and load with `open("static/app.css").read()`

**11. No `__all__` exports** in any `src/*.py` module
- No explicit public API; any internal symbol is importable
- Low priority but good hygiene

**12. `print_raster_info` has print side-effect** (`utils.py:200`)
- A utility function that always prints to stdout is surprising behavior
- Fix: return the dict, let callers print if needed; or add `verbose=True` param

**13. UTM EPSG arithmetic is non-obvious** (`utils.py:65`)
```python
epsg_code = hemisphere * 100 + zone_number   # e.g. 326*100+1 = 32601 ✓
```
Correct but fragile-looking. A comment explaining the EPSG numbering scheme (WGS84 UTM North = 326xx) would prevent future maintainer confusion.

**14. `AUDIT_REPORT.txt` documents a resolved issue as a bug**
- The Feb 2026 audit flagged `performance_ratio` as "never applied — CRITICAL BUG"
- This was resolved via documentation (monthly_generation_kwh_per_kw is a DELIVERED value)
- `AUDIT_REPORT.txt` is now stale and misleading; this `CONTEXT.md` supersedes it
